Cross origin communication

SOP (Same Origin Policy) security measure of a browser is a restrictive feature and by default blocks any communication between different [[origin]]. But there are cases when different origins need to legitimately communicate with each other.

In such cases, developer essentially has 2 options (adding cors exception or using postMessages):

1. CORS

Explicity allow different origin access to the site using following headers. It can be done by adding the site (b.com) to the access control list. Some sites balantly reflect the origin in access control header or have a wildcard * or *.example.com.

  • Access-Control-Allow-Origin: https://siteb.com
    Allows site B javascript to read the response. It can also be an wildcard (*) allowing any origin to read the response.

  • Access-Control-Allow-Credentials: true
    When the cookies or auth header or TLS cert is sent with request, the browser allows the response to be read by the cross origin only if it is allowed in the Access-Control-Allow-Origin (can’t be a wildcard( *)) and Access-Control-Allow-Credentials is set to true in the response. This header is meaningless if no cookies or auth headers are included in the request.

2. postMessage

postMessage is the browser’s intentional API for cross-origin communication between windows/tabs/iframes.

Basic Mechanism postMessage:

  • Sending side
    Here the website a.com is sending a message hello to window which is opened inside the iframe. Here the postMessage is the property of contentWindow so we need to reference it. The browser first checks if the destination b.com exists if it doesn’t the request is dropped. If the destination exists then the post message i sent.

  • Receiving side
    The receiving window should have a listener set up for listening for postMessage.

Dangers in postMessage

  • Here the listener has a handler which doesn’t validate the origin. If handler does something dangerous like eval with the data then it might be vulnerable to attacks.
  • Sending to * targetOrigin with sensitive data. If attacker can get their iframe loaded in place of the legitimate one, they receive the secret.

Why is it dangerous?

Because SOP and CORS only restrict XMLHttpRequest and fetch, script tags are different story. The browser treats the request as loading any normal external javascript and doesn’t enforce any cross origin security measures. JSONP exploits this gap by sending data as js.

What makes it even more dangerous is that since the script tag is essentially making a GET request, the cookies are sent automatically. So, if evil.com has script tag which fetches from a.com then the request will include cookies of a.com meaning the server at a.com thinks its authenticated user is making request but in reality its the evil.com.

Also, since there is a reflection (the function name), if the server doesn’t sanitize it properly XSS might be possible.
